# 事前マッチ＋起こし合い機能：ロジックと制約の解説

対象ブランチ：`feature/prematch-wakeup`
この資料は「①どう動くように作ったか（ロジック）」と「②なぜ現段階では想定通りに動かないのか（制約と理由）」をまとめたもの。発表・質疑用。

---

## 0. この機能でやりたいこと（理想の姿）

1. **同じアラーム時刻に設定した人同士**を、アラームの**1時間前**にマッチさせて相手を確定する。
2. アラーム時刻に各自が**アラームを止める**と、トーク待機画面に進む。
3. 相手がまだ起きていなければ「**起きて！**」ボタンを押して、**相手のスマホに通知を鳴らして起こす**。
4. **2人とも起きたら**トークルームに移動して5分間チャットする。

---

## 1. 実装したロジック（現状の作り）

### 1.1 全体フロー

```
[アラーム1時間前〜アラーム時刻]
   同じ alarmTime の2人を Firestore トランザクションでマッチ
   → rooms/{roomId} 作成・両者 currentRoomId 設定
   → ホームに「🎉 マッチ成立！」
        ↓
[アラーム時刻]  通知で起床 → 「アラームを止める」
        ↓
   markAwake：room.awake[自分]=true にする
   → トーク待機画面へ移動
        ├─ 相手が未起床 → 「起きて！」→ room.wakePing に合図を書く
        │        → 相手アプリが検知 → 相手端末でローカル通知
        ↓
[2人とも awake=true]  markAwake が sessionStartedAt をセット
   → 両者のトーク待機画面がそれを検知 → トークルーム（5分チャット）へ
```

### 1.2 データ構造（Firestore）

| 場所 | フィールド | 役割 |
|---|---|---|
| `matching_pool/{uid}` | `alarmTime`, `joinedAt` | 待機列。**同じ alarmTime** の人を探すために時刻も持つ |
| `rooms/{roomId}` | `participants`[2], `names` | 参加者と名前（roomId は2人のUIDを並べた固定ID） |
| 〃 | `alarmTime` | 共有の起床時刻 |
| 〃 | `awake: { uidA:false, uidB:false }` | 誰がアラームを止めた（起きた）か |
| 〃 | `wakePing: { to, at }` | 「起きて！」の合図（相手が検知して通知を鳴らす） |
| 〃 | `sessionStartedAt` | 2人とも起きた瞬間の時刻（5分タイマーの基準） |

### 1.3 各ステップの詳細

**① 事前マッチ（アラーム1時間前）** … `lib/matching.ts` の `joinMatchingPool` / `tryMatch`
- ホームの1秒ごとのタイマーが「アラームまで残り60分以内か」を判定。
- 該当したら `matching_pool` に自分＋alarmTime を登録し、**同じ alarmTime の相手**を探す。
- 見つかったら**トランザクション**で `rooms` を作成し、両者の `currentRoomId` に部屋IDを書く。
- 各自は自分の `users` を `onSnapshot` で監視していて、`currentRoomId` が入ると「マッチ成立」を検知する。

**② アラームを止める** … `markAwake(uid, roomId)`
- 部屋の `awake[自分]=true` にする（トランザクション）。
- その時点で**2人とも true** なら `sessionStartedAt` をサーバー時刻でセットする。
- 押した本人はトーク待機画面へ移動する。

**③ 「起きて！」** … `sendWakePing(roomId, 相手uid)`
- 部屋の `wakePing = { to: 相手uid, at: 今の時刻 }` を書く。
- **相手のアプリ**（ホームで待ち受け中）が `onSnapshot` でこの変化を検知し、
  「自分宛て・新しい合図」なら**自分の端末でローカル通知**「早く起きてください。トーク相手はすでに起きています。」を鳴らす。

**④ 2人とも起床 → トークルーム**
- `sessionStartedAt` がセットされると、両者のトーク待機画面が `onSnapshot` で検知し、
  チャット画面へ `replace` 遷移する。5分タイマーはこの `sessionStartedAt` から始まる。

### 1.4 主要関数（`lib/matching.ts`）
- `joinMatchingPool(uid, alarmTime)` … 待機列に参加
- `tryMatch(uid, alarmTime)` … 同時刻の相手とペア成立（トランザクション・楽観ロック・最大5回）
- `markAwake(uid, roomId)` … 起床を記録、2人揃えば開始時刻セット
- `sendWakePing(roomId, targetUid)` … 「起きて！」の合図を書く

## 2. なぜ現段階では想定通りに動かないのか（制約と理由）

結論：**「寝ている相手を起こす」「1時間前に全員を自動でマッチ」は、今の構成（Expo Go＋Firebase無料プラン＋サーバー無し）では原理的にできない。** 現状は「両方の端末をアプリ起動中にしたデモ」でのみ成立する best-effort 実装。

### 2.1 前提となっている3つの制約

このプロジェクトのルール：
- **Expo Go** で動かす（開発ビルドは使わない）
- **Firebase 無料（Spark）プラン** … Cloud Functions（サーバー処理）は使えない
- つまり **サーバー側の処理が一切無く、全部アプリ（クライアント）側で動く**

この前提から、次の3つの「できないこと」が生まれる。

### 2.2 制約A：アプリが前面に無いと、中身（JavaScript）が止まる

スマホのOSは、前面に出ていないアプリの処理をバッテリー節約のため**サスペンド（凍結）**する。

| スマホの状態 | アプリのJS（マッチ判定・相手監視・「起きて！」受信） | 予約通知（アラーム本体） |
|---|---|---|
| アプリを前面に表示中 | ✅ 動く | ✅ 出る |
| 画面を消した（ロック） | ❌ 数秒で停止 | ✅ 出る |
| 他のアプリを使用中 | ❌ 数秒で停止 | ✅ 出る |
| アプリを完全終了 | ❌ 停止 | ✅ 出る |

**「起きて！」の受信はアプリのJSが担う**ので、相手のスマホが「画面OFF・他アプリ・終了」だと受け取れない。
→ ところが**寝ている人のスマホは、まさに画面OFF**。だから「寝ている相手を起こす」が成立しない。

（右端の「予約通知」だけはOS自身が配信するので、アプリが止まっていても出る。＝自分のアラームは鳴る。）

### 2.3 制約B：プッシュ通知が使えない（Expo Go）

「相手のスマホに、アプリが止まっていても通知を出す」には、**プッシュ通知**（サーバーから相手のOSへ直接送る通知）が必要。

| | ローカル通知（今使っている） | プッシュ通知（本当は必要なもの） |
|---|---|---|
| 誰が出す | 自分の端末のアプリ | サーバー → 相手のOS |
| アプリ停止中でも届く | ❌ | ✅ |
| Expo Go で使える | ✅ | ❌（開発ビルドが必要） |

**プッシュ通知の受信は Expo Go ではサポート外**（SDK 53以降）。受け取るには「開発ビルド」が必要で、それはこのプロジェクトのルール（Expo Go縛り）から外れる。
→ だから「起きて！」は、相手の端末が**アプリを開いている時のローカル通知**でしか実現できない。

補足：**費用の問題ではない**。FCM（Firebaseの通知）もExpoの通知も送信自体は無料。ネックは「Expo Goでは受信できない＝開発ビルドが要る」という点。

### 2.4 制約C：「1時間前に全員を自動でマッチ」できない

「特定の時刻に、その場にいない人も含めて処理を走らせる」には**サーバー（Cloud Functions）**が必要。無料プランでは使えない。
→ 現状は「アラームの1時間前〜時刻の間に**アプリを開いていれば**、その時にマッチ処理が走る」という形。全員が寝ている（アプリを開いていない）1時間前に、自動で裏でマッチ、はできない。

### 2.5 理想 vs 現状の対応表

| 理想の動作 | 現状（デモ版）でできること | できない理由 |
|---|---|---|
| 1時間前に自動で同時刻の人をマッチ | その時間帯に**アプリを開けば**マッチ | サーバー処理が無い（制約C） |
| 相手がアラームを止めたら分かる | ✅ できる（Firestoreで共有） | — |
| 「起きて！」で**寝ている相手**を起こす | **相手がアプリを開いている時だけ**通知 | 背景でJS停止＋プッシュ不可（制約A・B） |
| 2人とも起きたらトークへ | ✅ できる | — |

## 3. 本番で実現するには（将来の拡張）

「アプリを閉じている相手も起こす」を本当に実現するには：

1. **開発ビルドに移行**（Expo Go を卒業）… 無料だが初回セットアップが必要
2. **プッシュ通知を導入**（Expo Push もしくは FCM）… 相手のOSへ直接届く。送信は無料
3. **送信のトリガー**：厳密には Cloud Functions（要 Blaze プラン・従量課金だが無料枠あり）を挟むのが安全。
   簡易にはクライアントから Expo Push API を叩く方法もある（無料）。

つまり技術的には無料でも可能だが、「**Expo Go をやめて開発ビルドにする**」という方針転換が必要になる。

## 4. 発表での説明ポイント（想定Q&A）

- **Q. なぜ「起きて！」が相手に届かないことがある？**
  A. 寝ている相手のスマホは画面OFFで、アプリの処理が停止しているから。ローカル通知はアプリが動いている時しか出せず、閉じている相手に届けるにはプッシュ通知（開発ビルドが必要）がいる。

- **Q. お金を払えば解決する？**
  A. 通知の送受信自体は無料。ネックはお金ではなく「Expo Go ではプッシュ通知を受け取れない＝開発ビルドが必要」という点。

- **Q. 1時間前のマッチは自動？**
  A. サーバーが無いので完全自動ではなく、その時間帯にアプリを開いていれば成立する best-effort。

- **Q. なぜこの制約を承知で作った？**
  A. まず Expo Go で「仕組みとUXが成立すること」をデモで示し、本番は開発ビルド＋プッシュ通知が必要、と切り分けて設計しているため。制約を理解した上での段階的な作りになっている。
